# Product Requirement Workflow

这是一套**完整闭环的产品需求协作流程**，不依赖 `/brainstorming` 或其他 skill。它从一句话需求开始，通过内置需求访谈、PRD/Markdown、HTML 交互说明、产品自审、开发准入、mock 确认、范围冻结和变更控制，尽量把产品与开发之间的反复修改前置到产品自审阶段。

## 一句话目标

让产品先和 AI 把需求聊透、文档写清、交互说明看明白、自己多轮自审，再把已确认的范围交给开发，减少开发完成后才发现“不是我想要的”的返工。

## 适用场景

- 产品只有一句粗略需求，需要逐步澄清成可开发文档。
- 开发想让产品先确认需求范围，再进入 mock 或真实接口开发。
- 用户希望生成 PRD/Markdown、HTML 说明页、产品自审清单和开发准入结论。
- 用户提到“需求澄清”“产品确认”“减少返工”“HTML 交互说明”“mock 确认”“范围冻结”“变更控制”。
- 已有 Markdown 需求文档，但需要检查是否足够进入开发，或需要生成产品自审用 HTML。

## 不适用场景

- 用户只想简单改代码、修 bug、调样式。
- 用户明确要求跳过需求确认直接开发。
- 用户只需要普通 Markdown 转 HTML 预览，不涉及需求闭环。

## 闭环流程总览

```text
0. 一句话需求
   ↓
1. AI 需求访谈
   ↓
2. 需求澄清摘要
   ↓
3. PRD / Markdown 需求文档
   ↓
4. HTML 交互说明文档
   ↓
5. 产品自审与多轮修改
   ↓
6. 开发准入检查
   ↓
7. Mock 页面开发与确认
   ↓
8. 范围冻结
   ↓
9. 真实接口开发
   ↓
10. 变更记录 / 二期池
```

## 核心规则

1. **不要直接从一句话需求进入开发。**
2. **不要只把 Markdown 美化成 HTML。** HTML 必须承担产品自审和交互确认作用。
3. **不要假装需求已经清楚。** 不清楚的内容必须进入“未决问题”。
4. **不要让口头新增需求混进开发。** 产品确认后的新增内容必须进入“变更记录”。
5. **不要在 mock 阶段纠缠真实接口。** mock 只确认页面结构、字段、按钮、状态和交互闭环。
6. **不要在真实接口阶段随意改范围。** 范围冻结后只接受轻微文案/样式调整；流程、字段、页面、状态变动需要重新评审。

## 阶段 0：接收一句话需求

### 输入示例

```text
要做一个盘点管理。
要新增客户回访功能。
需要一个工单质检模块。
```

### 第一反应

先不要写 PRD，也不要开发。应该输出简短确认：

```text
我先按完整需求闭环来澄清：目标、角色、范围、页面、流程、状态、字段、权限、异常和验收标准。我们先确认会影响开发结构的关键问题。
```

然后进入阶段 1。

## 阶段 1：AI 需求访谈

这是本 skill 内置的 brainstorming 阶段，不调用 `/brainstorming`。

### 访谈方式

- 每轮只问 3-6 个关键问题，避免一次性抛出长问卷。
- 先问会影响架构和开发范围的问题，再问字段细节。
- 用户回答后，及时总结已确认内容和仍未确认内容。
- 对于用户没说但业务上必须明确的内容，给出推荐默认值，并标注“可调整”。

### 问题优先级

#### 第一轮：目标与边界

必须确认：

1. 这个模块解决什么业务问题？
2. 本期必须完成哪些能力？
3. 哪些能力明确不做或放到二期？
4. 谁是主要使用人？
5. 最终以什么结果证明这个模块完成？

#### 第二轮：角色与流程

必须确认：

1. 谁创建数据？
2. 谁处理流程？
3. 谁确认或审批结果？
4. 谁只能查看？
5. 从创建到结束的主流程是什么？
6. 是否存在取消、驳回、撤回、重复提交、重复确认等场景？

#### 第三轮：页面与交互

必须确认：

1. 需要哪些页面：列表、详情、新增/编辑、处理页、确认弹窗、导入导出？
2. 新增和编辑是否共用表单？
3. 详情是否只读？
4. 哪些操作用抽屉，哪些操作用弹窗，哪些需要独立页面？
5. 哪些字段需要联动？
6. 哪些字段只在特定状态可见或可编辑？

#### 第四轮：字段与数据

必须确认：

1. 查询条件有哪些？
2. 列表字段有哪些？
3. 表单字段有哪些？
4. 详情字段有哪些？
5. 哪些字段必填？
6. 哪些字段系统生成？
7. 哪些字段后端计算？
8. 哪些字段来自字典或枚举？
9. 是否涉及 Long ID，前端是否必须保持字符串？

#### 第五轮：状态、权限、异常、验收

必须确认：

1. 有哪些状态？
2. 状态如何流转？
3. 每个状态允许哪些按钮？
4. 每个角色能看到哪些按钮？
5. 哪些情况必须阻止提交？
6. 哪些重复操作应幂等返回？
7. 错误提示文案要说明什么？
8. 测试如何判断功能通过？

### 每轮访谈输出格式

```markdown
## 本轮已确认

- xxx
- xxx

## 当前推荐默认值

- xxx：建议 xxx，原因 xxx。

## 仍需确认

1. xxx？
2. xxx？

## 下一轮问题

1. xxx？
2. xxx？
```

## 阶段 2：需求澄清摘要

当关键问题基本明确后，先输出“需求澄清摘要”，不要立刻写长 PRD。

### 摘要模板

```markdown
# 需求澄清摘要

## 1. 模块目标

## 2. 本期范围

## 3. 明确不做

## 4. 用户角色与职责

## 5. 页面清单

## 6. 主流程

## 7. 状态设计

## 8. 字段清单

## 9. 操作与权限

## 10. 异常场景

## 11. 验收标准

## 12. 未决问题

## 13. 推荐进入下一步

- 是否建议生成 PRD：是 / 否
- 阻塞项：
  1. xxx
```

### 进入 PRD 的条件

只有满足以下条件，才建议生成 PRD：

- 模块目标明确。
- 本期范围明确。
- 不包含范围明确。
- 主角色明确。
- 至少有初步页面清单。
- 至少有初步主流程。
- 未决问题不影响整体文档骨架。

如果未满足，继续访谈，不写 PRD。

## 阶段 3：生成 PRD / Markdown 需求文档

### 文件定位

- 默认放在对应业务目录的 `docs/` 下。
- 文件名使用业务模块中文名，例如 `盘点管理.md`。
- 如果用户提供了具体目录，使用用户目录。
- 不覆盖已有文档，除非用户明确要求更新。

### PRD 结构

```markdown
# 模块名称设计

## 1. 模块目标

## 2. 业务范围
### 2.1 包含
### 2.2 不包含

## 3. 角色职责

## 4. 菜单与页面结构
### 4.1 列表页
### 4.2 新增 / 编辑
### 4.3 处理 / 录入 / 确认页面
### 4.4 详情页

## 5. 核心流程

## 6. 状态设计

## 7. 字段设计

## 8. 业务规则

## 9. 接口建议

## 10. 异常处理

## 11. 前端实现要点

## 12. 与现有模块对齐项

## 13. 验收标准

## 14. 产品自审清单

## 15. 开发准入清单

## 16. 未决问题

## 17. 变更记录
```

### 字段表规范

涉及数据库表、接口模型或页面字段时，字段表必须包含：

| 字段名 | 中文名 | 字段类型 | 是否必填 | 说明 |
| --- | --- | --- | --- | --- |

规则：

- `说明` 中涉及典型取值时必须写“示例：xxx”。
- 系统生成字段必须标明。
- 后端计算字段必须标明。
- 前端只读字段必须标明。
- 字典字段必须标明字典类型或枚举值。
- 后端 Long ID 前端保持 `string` 或项目内 `ID` 类型，不做数值化转换。

### PRD 质量门槛

合格 PRD 必须具备：

- 有“不包含范围”。
- 有“状态设计”。
- 有“异常处理”。
- 有“验收标准”。
- 有“产品自审清单”。
- 有“开发准入清单”。
- 有“未决问题”。
- 有“变更记录”。

缺一项都需要补齐。

## 阶段 4：生成 HTML 交互说明文档

### HTML 定位

HTML 是产品自审材料，不是普通文档预览。它应该让产品看到：

- 这个模块有哪些页面。
- 每个页面负责什么。
- 每个页面有哪些字段和按钮。
- 流程从哪里开始，到哪里结束。
- 状态如何变化。
- 哪些情况会被拦截。
- 哪些内容已经确认，哪些还没确认。

### 输出约定

- 默认与 PRD 同目录，同名 `.html`。
- 单文件自包含：内联 CSS，不依赖 CDN、外部图片、外部字体或外部 JS。
- 不依赖 Mermaid，流程图用 HTML/CSS 表达。
- 可本地双击打开。

### HTML 必备区块

1. **封面 / 模块总览**
   - 模块名称。
   - 业务目标。
   - 适用角色。
   - 本期范围摘要。

2. **确认状态条**
   - 文档版本。
   - 当前状态：草稿 / 待产品确认 / 已确认 / 已冻结。
   - 未决问题数量。
   - 最近变更。

3. **关键能力卡片**
   - 3-6 个核心能力。
   - 每个能力说明其业务价值。

4. **业务闭环流程图**
   - 创建、处理、提交、确认、完成等主路径。
   - 标注哪个动作真正影响业务结果。
   - 标注哪些状态可回退、取消或不可逆。

5. **页面地图**
   - 列表页。
   - 新增/编辑抽屉。
   - 处理页或处理抽屉。
   - 确认弹窗。
   - 详情页。
   - 关联页面或流水页。

6. **页面具象说明**
   每个页面必须包含：
   - 页面职责。
   - 进入条件。
   - 查询条件。
   - 列表字段。
   - 表单字段。
   - 操作按钮。
   - 按钮显示条件。
   - 字段联动。
   - 校验规则。
   - 空状态。
   - 异常状态。

7. **状态机**
   - 状态卡片。
   - 状态流转图。
   - 每个状态允许的操作。
   - 不可逆状态高亮。

8. **字段与规则**
   - 查询字段、列表字段、表单字段、详情字段分组展示。
   - 必填、只读、系统生成、后端计算字段视觉区分。

9. **权限矩阵**
   - 角色 x 操作。
   - 明确新增、编辑、删除、提交、确认、取消、导出、查看等权限。

10. **异常场景表**
    - 场景。
    - 触发条件。
    - 系统处理。
    - 提示文案。

11. **产品自审 Checklist**
    - 范围是否完整。
    - 不做项是否能接受。
    - 页面是否够用。
    - 字段是否够用。
    - 状态是否正确。
    - 权限是否正确。
    - 异常是否覆盖。
    - 验收标准是否可测。

12. **开发准入 Checklist**
    - PRD 是否完成。
    - HTML 是否完成。
    - 产品是否自审。
    - 未决问题是否清零或延期。
    - 页面、字段、状态、权限、异常、验收是否确认。

13. **未决问题与变更记录**
    - 未决问题单独列出。
    - 变更记录按版本列出。

### HTML 视觉规范

- 版心宽度建议 `1180px` 到 `1280px`。
- 使用后台系统风格：浅色背景、白色卡片、蓝色或靛蓝主色。
- 使用卡片、徽章、步骤条、表格、权限矩阵、Checklist。
- 长表格必须可横向滚动。
- 移动端基础响应式：多列网格降为单列。
- 不使用外部图标库，不使用在线资源。
- 不写复杂动画，避免影响阅读。

## 阶段 5：产品自审闭环

HTML 生成后，不直接进入开发。必须引导产品自审。

### 产品自审顺序

1. 模块目标是否正确？
2. 本期范围是否完整？
3. 不包含范围是否能接受？
4. 页面清单是否够用？
5. 主流程是否闭环？
6. 状态流转是否符合业务？
7. 查询、列表、表单、详情字段是否够用？
8. 操作按钮是否正确？
9. 角色权限是否正确？
10. 异常场景是否覆盖？
11. 验收标准是否可测试？
12. 未决问题是否已解决或延期？

### 产品自审输出模板

```markdown
# 产品自审结论

- 自审轮次：第 1 轮
- 是否通过：通过 / 不通过
- 是否允许进入开发 mock：是 / 否

## 必须修改

1. xxx

## 可以延期

1. xxx

## 新增问题

1. xxx

## 确认冻结内容

- 页面：已确认 / 未确认
- 字段：已确认 / 未确认
- 状态：已确认 / 未确认
- 权限：已确认 / 未确认
- 异常：已确认 / 未确认
- 验收标准：已确认 / 未确认
```

### 多轮修改规则

- 每轮修改都更新 PRD 和 HTML。
- 每轮修改都追加变更记录。
- 未决问题减少到 0，或明确延期，才允许进入开发准入检查。

## 阶段 6：开发准入检查

进入开发前，必须输出准入结论。

### 准入检查模板

```markdown
# 开发准入检查

| 检查项 | 状态 | 说明 |
| --- | --- | --- |
| PRD/MD 已完成 | 是/否 | |
| HTML 交互说明已完成 | 是/否 | |
| 产品已完成自审 | 是/否 | |
| 未决问题清零或延期 | 是/否 | |
| 页面清单已确认 | 是/否 | |
| 字段清单已确认 | 是/否 | |
| 状态流转已确认 | 是/否 | |
| 权限矩阵已确认 | 是/否 | |
| 异常处理已确认 | 是/否 | |
| 验收标准已确认 | 是/否 | |

## 结论

- 是否允许进入 mock 开发：是 / 否

## 阻塞项

1. xxx
```

### 准入规则

- 任何关键项为“否”，原则上不进入开发。
- 如果用户强制跳过，应明确提示：“跳过需求门禁，后续返工风险较高”。
- 允许进入 mock，不代表允许接真实接口。

## 阶段 7：Mock 页面开发确认

### 目标

mock 页面用于验证产品看得见的内容：页面、字段、按钮、状态、提示和主流程。

### Mock 阶段确认项

- 菜单入口是否正确。
- 查询条件是否够用。
- 列表字段是否够用。
- 新增/编辑表单是否符合预期。
- 处理页、抽屉或弹窗是否符合预期。
- 详情页是否符合预期。
- 按钮显示条件是否正确。
- 状态标签是否正确。
- 空状态和异常提示是否正确。
- 主流程是否能完整走通。

### Mock 确认模板

```markdown
# Mock 确认结论

- 是否通过：通过 / 不通过
- 是否允许进入真实接口开发：是 / 否

## 必改项

1. xxx

## 可延期项

1. xxx

## 范围冻结说明

- 页面是否冻结：是 / 否
- 字段是否冻结：是 / 否
- 状态是否冻结：是 / 否
- 权限是否冻结：是 / 否
```

## 阶段 8：范围冻结

mock 确认通过后，真实接口开发前必须冻结范围。

### 冻结内容

- 页面清单。
- 查询条件。
- 列表字段。
- 表单字段。
- 详情字段。
- 操作按钮。
- 状态流转。
- 权限矩阵。
- 异常处理。
- 验收标准。

### 冻结后处理规则

| 变更类型 | 处理方式 |
| --- | --- |
| 文案、提示语、列宽、轻微样式 | 可纳入当前迭代 |
| 新增字段 | 进入变更记录，评估是否纳入当前版本 |
| 新增按钮 | 进入变更记录，评估权限和接口影响 |
| 新增页面 | 默认进入二期，除非明确批准 |
| 修改状态机 | 回到产品自审，重新确认 PRD 和 HTML |
| 修改主流程 | 回到需求访谈或产品自审 |
| 修改接口契约 | 评估前后端影响，更新 PRD、HTML 和 mock |

## 阶段 9：真实接口开发

真实接口开发阶段以已冻结范围为准。

### 开发要求

- 按 PRD 字段和接口建议实现。
- 按 HTML 说明验证页面交互。
- 若发现需求与实现冲突，先记录为问题，不擅自改范围。
- 若产品提出新要求，先写入变更记录。

### 接口阶段不应再讨论

- 是否需要新页面。
- 是否需要新增核心状态。
- 是否重做主流程。
- 是否替换主要交互方式。

这些问题应在产品自审或 mock 阶段解决。

## 阶段 10：变更记录 / 二期池

### 变更记录模板


## 已有 Markdown 时的处理方式

如果用户提供已有 Markdown：

1. 先做开发准入检查，不要直接转 HTML。
2. 检查是否包含：目标、范围、不做项、角色、页面、流程、状态、字段、权限、异常、验收、未决问题、变更记录。
3. 缺失内容先列为补充问题。
4. 如果文档足够完整，再生成 HTML 交互说明。
5. HTML 生成后仍然要求产品自审，不直接进入开发。

## 直接开发请求的处理方式

如果用户要求直接开发：

- 若已有 PRD、HTML、自审结论和准入通过，可以开发。
- 若没有，应先提醒风险，并建议走闭环流程。
- 若用户明确要求跳过，继续执行用户要求，但在回复中说明：

```text
本次跳过需求门禁，后续产品确认阶段出现范围变更的风险较高。
```

## 最终产物清单

完整流程结束时，应至少产出：

1. `需求澄清摘要`
2. `模块名.md` PRD / 需求设计文档
3. `模块名.html` 产品自审交互说明文档
4. `产品自审结论`
5. `开发准入检查`
6. `Mock 确认结论`
7. `变更记录 / 二期池`

## 完成标准

一个模块可以进入低返工开发状态，必须满足：

- 产品能通过 HTML 看懂功能全貌。
- 产品能按 checklist 自己发现问题并修改。
- 开发能按 PRD 明确页面、字段、状态、接口和验收标准。
- 测试能按验收标准设计用例。
- 未决问题已清零或延期。
- mock 确认后范围已冻结。
- 真实接口开发阶段只处理实现问题，不重新讨论核心需求。
