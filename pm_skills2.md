# High Fidelity HTML Prototype

这是一套用于**高保真 HTML 交互原型**的规范化流程。它接在 `pm_skills2.md` 之后，用于把已澄清的 PRD / Markdown 需求文档转成接近真实后台系统的单文件 HTML 原型，让产品在 mock 开发前完成页面、字段、按钮、状态、弹窗和主流程自审。

## 一句话目标

在正式开发 mock 页面之前，用一个可双击打开的高保真 HTML 原型，把产品能看到、能点击、能评审的页面结构和交互流程提前暴露出来，减少开发完成后才反复改页面和字段的返工。

## 适用场景

- 已有 PRD / Markdown 需求文档，需要生成产品可审阅的高保真 HTML 原型。
- 已有 HTML 交互说明文档，但需要进一步变成接近真实后台页面的原型。
- 用户提到“高保真 HTML”“交互原型”“原型自审”“产品确认页面”“mock 开发前确认”。
- 后台管理系统页面较多，涉及列表、查询、抽屉、弹窗、状态流转、权限按钮、异常提示。
- 需要让产品在开发前确认：页面够不够、字段对不对、按钮是否符合状态、弹窗是否完整。

## 不适用场景

- 用户只需要普通 Markdown 转 HTML 文档。
- 用户只需要低保真流程说明，不需要接近真实页面的视觉结构。
- 用户已经明确要求直接开发业务代码。
- 需求还没有 PRD、页面清单、字段、状态、按钮和异常规则。

如果需求信息不足，不要直接生成高保真原型，应先回到 `product-requirement-workflow` 补齐需求。

## 与其他流程的关系

推荐链路：

```text
product-requirement-workflow
  -> PRD / Markdown 需求文档
  -> HTML 交互说明文档
  -> 产品确认需求边界
  -> high-fidelity-html-prototype
  -> 高保真 HTML 原型
  -> 产品原型自审
  -> mock 页面开发
  -> mock 确认
  -> 范围冻结
  -> 真实接口开发
```

本 skill 只负责高保真 HTML 原型，不负责真实 Vue / API / mock 接口开发。

## 核心规则

1. **不要从一句话需求直接生成高保真原型。** 必须先有 PRD 或足够完整的需求文档。
2. **不要把 HTML 原型当成普通文档排版。** 它必须具备真实后台页面的视觉结构。
3. **不要扩写未确认需求。** 文档未确认的内容必须标注为“待确认”，不能伪装成已确认。
4. **不要写业务代码。** 本流程只产出静态 HTML 原型。
5. **不要依赖外部资源。** HTML 必须单文件自包含，不能依赖 CDN、外部图片、外部字体或外部 JS。
6. **不要跳过产品自审。** 原型生成后必须输出产品原型自审清单和开发准入结论。
7. **不要一次性做过大范围。** 页面很多时，先做核心闭环原型，再分批补齐外围页面。

## 阶段 1：输入检查

生成高保真原型前，必须先检查输入材料。

### 必须读取

- PRD / Markdown 需求文档。
- 如存在，读取 HTML 交互说明文档。
- 如存在，读取对应模块已有子页面文档。
- 如涉及现有前端项目，读取 `CLAUDE.md` 或项目规则，确保风格和交互与项目一致。

### 输入准入条件

必须能从文档中提取到：

- 模块目标。
- 本期范围和不包含范围。
- 页面清单。
- 查询条件。
- 列表字段。
- 表单字段。
- 详情字段。
- 操作按钮。
- 状态流转。
- 权限或角色边界。
- 异常场景。
- 未决问题。

如果缺少关键内容，先输出“原型输入缺口”，不要生成 HTML。

### 原型输入缺口模板

```markdown
# 原型输入缺口

## 已具备

- xxx

## 缺少内容

| 缺口 | 影响 | 建议补充 |
| --- | --- | --- |
| xxx | xxx | xxx |

## 结论

- 是否允许生成高保真原型：是 / 否
- 阻塞项：
  1. xxx
```

## 阶段 2：原型范围选择

如果页面较多，不要默认一次性生成全部页面。先按业务闭环选择原型范围。

### 推荐优先级

1. 核心闭环页面：能完整走通主要业务流程的页面。
2. 高频操作页面：产品和业务每天会用到的页面。
3. 高风险页面：字段多、状态多、弹窗多、容易返工的页面。
4. 外围台账页面：只读或低频页面。

### 范围输出模板

```markdown
# 高保真原型范围

## 推荐本轮原型范围

1. xxx
2. xxx

## 暂不纳入本轮

1. xxx：原因 xxx。

## 原因

- xxx
```

如果用户没有特别指定，优先生成核心闭环原型。

## 阶段 3：视觉与交互规范

### 视觉风格

后台管理系统默认使用以下风格：

- 浅色背景。
- 白色卡片。
- 蓝色或靛蓝主色。
- 顶部模块说明区。
- 左侧或顶部页面导航。
- 查询区、工具栏、表格、分页、抽屉、弹窗、状态标签、提示条。
- 字体使用系统字体，不引入外部字体。
- 风格贴近 Ant Design Vue / Vben 管理端。

### HTML 输出约定

- 单文件 HTML。
- 内联 CSS。
- 不依赖 CDN。
- 不依赖外部 JS。
- 不依赖外部图片。
- 不依赖外部字体。
- 不使用构建工具。
- 可本地双击打开。
- 文件默认与 PRD 同目录，命名为 `模块名高保真原型.html`。

### 允许的轻量交互

可以使用少量内联 JavaScript，仅用于静态原型内的视觉切换，例如：

- Tab 切换。
- 页面锚点导航。
- 展开/收起。
- 模态框显示/隐藏。
- 抽屉示意显示/隐藏。

但必须满足：

- 不发起网络请求。
- 不读取本地文件。
- 不依赖外部库。
- 不模拟真实鉴权。
- 不写复杂业务逻辑。

如果不需要点击切换，优先不用 JavaScript，用静态区块表达。

## 阶段 4：页面原型必备结构

每个核心页面至少包含以下区块。

### 4.1 页面头部

- 页面名称。
- 页面职责。
- 当前确认状态：待确认 / 已确认 / 有未决问题。
- 关联角色。
- 关联状态。

### 4.2 查询区

- 展示所有查询字段。
- 标明控件类型，例如 Input、Select、DateRange、Cascader、Segmented。
- 标明默认值或默认筛选范围。
- 标明字段联动规则。

### 4.3 工具栏

- 新增、导出、批量操作等按钮。
- 标明按钮权限。
- 标明按钮显示条件。

### 4.4 表格区

- 展示核心列表字段。
- 至少提供 3-5 条示例数据。
- 状态字段用标签展示。
- 时间、金额、面积、联系方式按业务规则展示。
- 行操作必须按状态展示。

### 4.5 抽屉 / 弹窗区

涉及新增、编辑、详情、确认、流转时，必须在原型中具象化：

- 新增/编辑抽屉。
- 详情抽屉。
- 跟进记录弹窗。
- 状态流转弹窗。
- 确认弹窗。
- 标记原因弹窗。

每个抽屉或弹窗需要展示：

- 标题。
- 字段。
- 必填标识。
- 只读字段。
- 默认值。
- 底部按钮。
- 校验提示。

### 4.6 状态与按钮规则

每个页面必须明确：

- 当前状态有哪些。
- 每个状态允许哪些按钮。
- 哪些状态按钮隐藏。
- 哪些操作不可逆。
- 哪些操作需要二次确认。

### 4.7 空状态和异常状态

每个核心页面必须展示或说明：

- 无数据空状态。
- 查询无结果。
- 无权限状态。
- 非法状态流转提示。
- 重复提交 / 重复确认提示。
- 必填校验提示。

### 4.8 产品自审清单

每个核心页面后面附带该页面的自审清单：

- 查询字段是否够用。
- 列表字段是否够用。
- 操作按钮是否正确。
- 抽屉/弹窗字段是否完整。
- 状态限制是否符合业务。
- 异常提示是否清楚。
- 是否允许进入 mock 开发。

## 阶段 5：原型全局必备区块

高保真 HTML 原型必须包含以下全局区块。

### 5.1 封面 / 模块总览

- 模块名称。
- 业务目标。
- 本轮原型范围。
- 暂不纳入范围。
- 适用角色。
- 文档版本。
- 当前确认状态。

### 5.2 页面地图

用卡片、侧边导航或 Tabs 展示本轮包含的页面。

### 5.3 核心流程

用 HTML/CSS 绘制主业务流程，不使用 Mermaid。

### 5.4 状态机摘要

展示状态、阶段、允许操作和终态。

### 5.5 页面原型区

逐页展示高保真页面。

### 5.6 权限矩阵

展示角色 x 操作，至少覆盖新增、编辑、查看、导出、流转、确认、删除或关闭类动作。

### 5.7 异常场景

展示场景、触发条件、系统处理、提示文案。

### 5.8 原型自审清单

全局自审清单包括：

- 页面清单是否正确。
- 核心流程是否闭环。
- 查询字段是否完整。
- 列表字段是否完整。
- 表单字段是否完整。
- 弹窗/抽屉是否完整。
- 状态与按钮是否匹配。
- 权限是否匹配。
- 异常提示是否覆盖。
- 未决问题是否标注。

### 5.9 开发准入结论

明确输出：

- 是否允许进入 mock 开发：是 / 否。
- 仍需产品确认的问题。
- 本轮冻结的页面、字段、按钮、状态。
- 进入 mock 后不应再变更的内容。

## 阶段 6：高保真程度标准

### 合格标准

原型至少应做到：

- 页面布局接近真实后台系统。
- 查询区、表格、工具栏、分页、状态标签可视化。
- 抽屉和弹窗以真实结构展示。
- 示例数据能反映不同状态。
- 操作按钮按状态差异展示。
- 空状态和异常提示有具体文案。
- 产品能不看 PRD 也大致理解页面怎么用。

### 不合格表现

以下情况视为不合格：

- 只有文字说明，没有页面结构。
- 只有流程图，没有列表、表单、按钮、弹窗。
- 所有页面长得一样，没有状态差异。
- 未决问题没有标注，直接写成确定结论。
- 需要联网才能打开。
- 大量依赖外部库或在线资源。
- 与当前项目后台风格明显不一致。

## 阶段 7：产品原型自审输出

生成原型后，必须提示产品按以下模板自审。

```markdown
# 高保真原型自审结论

- 原型文件：xxx.html
- 自审轮次：第 1 轮
- 是否通过：通过 / 不通过
- 是否允许进入 mock 开发：是 / 否

## 必须修改

1. xxx

## 可以延期

1. xxx

## 新增问题

1. xxx

## 确认冻结内容

- 页面清单：已确认 / 未确认
- 查询字段：已确认 / 未确认
- 列表字段：已确认 / 未确认
- 表单字段：已确认 / 未确认
- 抽屉/弹窗：已确认 / 未确认
- 状态与按钮：已确认 / 未确认
- 权限：已确认 / 未确认
- 异常提示：已确认 / 未确认
```

## 阶段 8：多轮修改规则

- 每轮原型修改都追加变更记录。
- 每轮修改都同步更新原型内的版本、确认状态和未决问题数量。
- 如果修改只影响文案、字段顺序、列宽、轻微样式，可直接更新原型。
- 如果修改影响页面清单、主流程、状态机、权限矩阵或核心字段，应回到 PRD 更新，再重新生成原型。
- 产品确认通过后，输出开发准入结论。

## 文件命名规范

默认文件名：

```text
模块名高保真原型.html
```

如果已有同名文件，不要覆盖，除非用户明确要求更新。可使用版本号：

```text
模块名高保真原型-v1.1.html
```

## 质量自检

写完 HTML 后必须做以下检查：

1. 文件存在且非空。
2. 包含封面、页面地图、核心流程、状态机、页面原型、权限矩阵、异常场景、自审清单、开发准入结论。
3. 不包含英文或中文占位符，例如待办标记、未定标记、待补充。
4. 不包含网络协议地址、CDN、外部字体、外部图片资源。
5. 如果使用 `<script>`，只用于本地静态交互，不发起请求。
6. 表格在窄屏可横向滚动。
7. 移动端基础响应式可读。
8. 未决问题明确标注为待确认。

推荐使用脚本做最小校验：

```bash
python3 - <<'PY'
from pathlib import Path
html = Path('模块名高保真原型.html')
text = html.read_text(encoding='utf-8')
required = ['模块总览', '页面地图', '核心流程', '状态机', '权限矩阵', '异常场景', '自审清单', '开发准入']
missing = [item for item in required if item not in text]
forbidden = ['待补充', 'http://', 'https://', 'cdn']
hits = [item for item in forbidden if item.lower() in text.lower()]
print('missing:', missing or 'none')
print('forbidden:', hits or 'none')
if missing or hits:
    raise SystemExit(1)
PY
```

## 最终回复格式

完成后简短说明：

```markdown
已生成高保真 HTML 原型：`path/to/模块名高保真原型.html`。

已校验：单文件、自包含、无外部资源、必备区块完整。当前结论：进入产品原型自审，暂不建议直接进入 mock 开发，待产品确认页面、字段、状态、按钮、权限和异常提示后再冻结范围。
```

## 禁止事项

- 禁止在本 skill 中写 Vue、API、mock 后端或真实业务代码。
- 禁止自动创建提交，除非用户明确要求 commit。
- 禁止把未确认问题当作已确认事实。
- 禁止为了好看而新增 PRD 没有的业务功能。
- 禁止依赖外部 UI 框架、CDN、图片或字体。
- 禁止让高保真原型成为新的需求来源；需求变更必须回写 PRD。
